# 串扰对延迟的影响

## 一句话理解

电容串扰不仅会造成噪声，还会造成 data-dependent delay：相邻线同向翻转时耦合电容几乎不消耗驱动电流，反向翻转时耦合电容的电压摆幅变成两倍，等效负载显著增大，传播延迟随邻线活动模式变化。

## 受害线和攻击线

考虑三根并行线：

```text
X   aggressor
Y   victim
Z   aggressor
```

$Y$ 与 $X$、$Z$ 之间各有耦合电容 $C_c$。

若 $Y$ 的翻转方向与 $X$、$Z$ 相反，则每个耦合电容两端电压变化为两倍信号摆幅。

因此：

```text
每个 C_c 等效贡献约 2C_c 的负载
两个邻线合计贡献约 4C_c
```

## 等效负载电容范围

设 victim node $Y$ 对地电容为 $C_{GND}$，每侧耦合电容为 $C_c$，则等效负载电容会随数据活动变化。

教材给出的范围直觉是：

$$
C_{GND}\le C_L\le C_{GND}+4C_c
$$

其中：

- 下界：邻线与 victim 同向翻转，耦合电容两端电压基本不变，贡献近似为 0。
- 上界：两侧邻线与 victim 反向翻转，每个耦合电容贡献约 $2C_c$。

## Miller effect 的角色

这里的“反向翻转导致等效电容加倍”就是熟悉的 Miller effect。

在深亚微米密集布线中，coupling capacitance 往往占总电容的较大比例，因此 Miller effect 会显著改变延迟。

## data-dependent delay

由于 $C_L$ 取决于邻线活动模式，传播延迟也变成数据相关：

```text
同向翻转：延迟小
反向翻转：延迟大
非同时翻转：取决于精确时序
```

这给 timing verification 带来困难，因为工具不仅要知道逻辑值，还要知道邻线翻转的相对时间。

## 为什么不能简单 always worst-case

若保守地总是假设最坏 Miller effect：

```text
C_L = C_GND + 4C_c
```

可以保证安全，但可能过度悲观，导致：

- 无谓加大 gate / buffer。
- 更大面积。
- 更高动态功耗。
- 反而可能加重供电噪声。

因此实际设计中需要在安全性和悲观度之间折中。

## 三类处理方法

### evaluate and improve

流程：

1. 完成布线。
2. 提取寄生参数。
3. 运行详细时序/噪声仿真。
4. 找到 bottleneck。
5. 修改版图或电路。

优点：通用、符合常规 EDA 流程。  
缺点：迭代慢，后期返工成本高。

### constructive layout generation

让 routing tool 在布线时主动考虑邻线串扰和延迟影响。

优点：更早处理问题。  
缺点：工具复杂，约束多，难以完全可靠自动化。

### predictable structures

采用预定义、规则、保守的布线结构，使延迟和串扰更可预测。

例子：

- FPGA 风格互连网格。
- dense wire fabric。
- 同层 signal 之间插入 $V_{DD}$ / GND shield。
- 相邻层正交布线。

教材提到 dense wire fabric 可把串扰几乎消除，并把 delay variation 降到很小，但代价是面积、电容、性能和功耗开销。

## 和 STA 的关系

串扰延迟会让静态时序分析更复杂，因为：

- 等效负载不是常数。
- 邻线方向和时序会改变受害线延迟。
- clock、reset、bus 等长线同时是性能路径和噪声源。

因此在先进工艺中，需要 coupling-aware extraction 和 SI-aware timing。

## 设计直觉

- crosstalk 不只是 noise issue，也是 timing issue。
- 最坏串扰通常发生在 victim 与两侧 aggressor 反向同时翻转。
- 同向翻转可能让耦合电容贡献接近 0。
- 用规则布线换可预测性，是复杂设计中常见的工程折中。
- 过度追求最坏情况会牺牲面积和功耗。

## 易错点

- 不要把线电容当成固定的 $C_L$；密集布线中 $C_L$ 与邻线活动相关。
- 不要只检查串扰噪声峰值，也要检查串扰导致的延迟变化。
- 不要无脑把所有耦合电容乘 2；需要看 victim 与 aggressor 的相对翻转方向。
- 不要把 shielding 当成无代价优化；它会增加对地电容。
