# 时序问题章节总览

## 一句话理解

第 10 章讨论数字电路里“事件何时发生、数据何时有效、什么时候采样”的问题。同步设计用全局 clock 给所有事件排序，但 clock skew 和 clock jitter 会让“同时发生”和“固定周期”这两个假设不再理想，从而限制性能并可能导致功能错误。

## 为什么需要时序方法

所有时序电路都有一个共同要求：

```text
必须强制规定 switching events 的先后顺序。
```

否则可能发生：

- 新数据过早覆盖旧数据。
- 存储单元采到错误值。
- 组合逻辑尚未稳定就被采样。
- 不同模块之间出现 race condition。
- 跨时钟域或异步输入导致 metastability。

## 同步设计的核心假设

同步系统使用全局周期性同步信号：

```text
global clock
```

所有 memory elements 在 clock 边沿附近更新。这样可以把复杂事件排序简化为：

```text
上一拍采样 -> 组合逻辑计算 -> 下一拍采样
```

这种方法简单、结构化、易验证，因此非常流行。

## 同步设计的代价

同步设计依赖两个理想化假设：

1. 全芯片 clock 到达所有寄存器的时间相同。
2. clock 周期完全固定。

实际中这两个假设都不成立：

- 空间上的 clock arrival time 差异叫 clock skew。
- 时间上的 clock period / edge 变化叫 clock jitter。

这些不确定性会：

- 降低可用计算时间。
- 压缩 setup margin。
- 压缩 hold margin。
- 造成 race condition。
- 限制最高频率。

## 设计直觉

- clock 是同步系统里的“全局裁判”，但裁判信号自身也有延迟和噪声。
- setup 问题常常可以通过降频缓解；hold/race 问题通常不能靠降频解决。
- 正 skew 可能看似提高性能，但会增加 hold failure 风险。
- jitter 永远吃掉时序裕量。
- clock network 通常是高性能芯片里最重要、最耗电、最难验证的网络之一。
- 异步/自定时设计能避开全局 clock，但会引入握手、完成检测和电路复杂度。
